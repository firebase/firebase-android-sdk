# Rewire Bidirectional gRPC Connection to GrpcBidiFlow

## Context

The Data Connect SDK manages a bidirectional gRPC stream to subscribe to realtime queries.
Historically, the gRPC library's coroutine stubs (e.g., `ConnectorStreamServiceCoroutineStub`) expose bidirectional streaming methods that accept a `Flow<Request>` and return a `Flow<Response>`.

While elegant, this design poses a severe architectural challenge: **it is extremely difficult to reliably correlate the lifecycle of the request flow with its corresponding response flow**.
Because of this limitation, we were unable to use advanced sharing strategies like `SharingStarted.WhileSubscribed()` in `DataConnectBidiConnectStream.kt`. Instead, we were forced to use `SharingStarted.Lazily` or `SharingStarted.Eagerly`. However, `WhileSubscribed` is highly desirable because its semantics exactly match the reconnect and cleanup behavior required by the Data Connect SDK (automatically closing connections when no active query subscribers remain, and reopening them dynamically on new subscriptions).

Furthermore, the low-level gRPC coroutines implementation hides execution details under deep, nested internal abstractions. This makes it very difficult for developers to set breakpoints, observe connection state changes, or debug complex connection failures.

## Decision

We will rewire `DataConnectGrpcRPCs.connect` to use a custom `GrpcBidiFlow.create` utility instead of the standard coroutine stubs.

The core code of `GrpcBidiFlow.kt` was initially copied verbatim from `ClientCalls.kt` inside the gRPC Kotlin library and then adapted. Instead of accepting a request flow from the caller, the adapted `GrpcBidiFlow` takes responsibility for creating a `kotlinx.coroutines.channels.Channel` for the outgoing requests. It then "sends" this channel downstream to the collector as part of an `Event.Started` lifecycle event. This makes the correlation between the request stream and the response stream perfectly clear and reliable.

Using this new correlation model, we will modify `DataConnectBidiConnectStream` to use `SharingStarted.WhileSubscribed()` for sharing the response flow, ensuring robust, lazy, and self-cleaning connection behavior.

We also introduce a robust `Listener` interface in `GrpcBidiFlow` to enable fine-grained logging of connection events.

## Rationale

- **Lifecycle Correlation**: By having `GrpcBidiFlow` construct the request channel and pass it downstream upon connection startup, it resolves the challenge where a flow collector could not easily get the exact request channel corresponding to its own collection lifecycle. This reliable correlation enables the use of `SharingStarted.WhileSubscribed()`.
- **Observability and Comprehension**: The `GrpcBidiFlow` implementation inlines several necessary but previously deeply-nested abstractions from the gRPC codebase. This makes the connection pipeline far easier for humans to comprehend, and makes setting breakpoints in lower-level logic simple.
- **Fine-Grained Debugging**: The introduced `Listener` interface allows us to log highly detailed connection transitions and raw frame exchanges.

## Options Considered

* **Option A: Keeping direct stub usage and fixing channels**
  * *Pros:* Fewer changes to existing files; no need to introduce a new utility class.
  * *Cons:* Code remains very complex, hard to maintain, and lacks clean separation between the connection lifecycle and the application-level multiplexing logic.
  * *Reason for Rejection:* We wanted a clean architectural split where the gRPC stream behaves like a standard cold Flow, making it easier to compose and reason about using standard Flow operators.

## Consequences

* **Positive:**
  * Seamless integration of `SharingStarted.WhileSubscribed()`, which perfectly aligns connection lifecycles with active application subscriptions.
  * Vastly improved debugging and developer ergonomics.
  * Clean separation between protocol-level multiplexing (`DataConnectBidiConnectStream`) and connection-level stream lifecycle (`GrpcBidiFlow`).
* **Negative/Risks:**
  * Code maintenance overhead of maintaining a custom fork of `ClientCalls.kt` logic (`GrpcBidiFlow.kt`). However, this logic is highly stable and unlikely to change.

## Future Work

- **Verbose Diagnostic Logging**: The highly detailed logs generated by the new `LoggingGrpcBidiFlowListener` are extremely valuable for diagnosing complex, hard-to-reproduce customer connection issues in production. We should introduce a mechanism (e.g., a `verbose` log level or a selective configuration flag) allowing developers or customers to selectively enable this level of diagnostic detail on demand.
