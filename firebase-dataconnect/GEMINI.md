This directory is an Android library gradle submodule for Firebase Data Connect.

The source code is written entirely in Kotlin.

## Gradle Information

This gradle submodule's build file is `firebase-dataconnect.gradle.kts`
instead of the conventional `build.gradle.kts`.
All submodules follow a similar naming convention for their gradle build files.

The root gradle project is defined in the parent directory.

The gradle wraper, "gradlew", is also located in the parent directory.
Therefore, when running gradle commands from this directory,
you **MUST** use `../gradlew`.
Notably, do **NOT** use `./gradlew` or `./demo/gradlew`.

As a build performance optimization, **ALWAYS** specify `--configure-on-demand`
to the gradle wrapper to avoid unnecessarily configuring other gradle submodules.

For example, to run the "assembleDebug" task, use this command:
`../gradlew --configure-on-demand assembleDebug`

The "scripts" subdirectory defines many useful scripts for running gradle commands,
which may be used instead of executing the gradle wrapper directly.

## SDK Entry Points

The primary "brain" and entry point for the SDK is the `FirebaseDataConnect` class and its factory methods.
When investigating core logic or starting an implementation, begin here rather than searching the entire `src` tree.
The implementation of the `FirebaseDataConnect` interface is `FirebaseDataConnectImpl`.

## Directory Layout

* src
  * Source code for the Firebase Data Connect Android SDK, and its unit and instrumentation tests.
* androidTestutil
  * Android library gradle submodule with utility classes and functions useful during Android instrumentation tests.
* testutil
  * Android library gradle submodule with utility classes and functions useful during unit **and** Android instrumentation tests.
* connectors
  * Android library gradle submodule that defines both unit and Android instrumentation tests for the code in the "src" directory.
  * **Crucially**, these tests simulate a real-world customer implementation by using exclusively public APIs from the "src" directory.
  * These tests use the code generated by the Firebase CLI rather than accessing the public APIs directly.
* ci
  * Scripts used by Github Actions workflows for continuous integration.
* demo
  * A simple demo application that uses the Firebase Data Connect SDK, as defined in the "src" directory.
* emulator
  * A Firebase project directory, as recognized by the Firebase CLI.
  * Defines Data Connect "schema" and "connectors" for use in unit and instrumentation tests.
  * Defines a helper script, `emulator.sh`, which launches a Postgresql server in a Podman container
    and the Data Connect emulator to use it.
* google-services.json
  * Defines fake Firebase project information, to facilitate testing against the Data Connect emulator.
* gradleplugin
  * Defines a custom gradle plugin and KSP kotlin compiler plugin.
  * The Gradle plugin automatically performs the code generation step during builds.
  * The KSP kotlin compiler plugin generates `codebase-map.md` files to help AI agents resolve imports in Kotlin source code.
* scripts
  * Defines several convenience shell scripts to run gradle commands.
  * These are intended for use by humans, but are absolutely available for AI agents to use as well.
  * These scripts are designed to be run from _any_ directory.
  * Use these scripts instead of directly running gradle commands, when appropriate:
    * compile_kotlin.zsh - Compile all Kotlin source code (including tests) and refresh the codebase-map.md files
    * run_unit_tests.zsh - Run all unit tests
    * run_integration_tests.zsh - Run all instrumentation tests.
    * spotlessApply.zsh - Format Kotlin code.

## Kotlin Import File Resolution

To efficiently find the source file for "import" statement in Kotlin files
for the com.google.firebase.dataconnect package and subpackages,
the Gradle build produces `codebase-map.md` files.

If you need to find the definition of something from the dataconnect
package (or one of its subpackages) then you **SHOULD** use these files,
if they exist, instead of crude methods like "grep" and "find" commands.

To generate these files, run ./scripts/compile_kotlin.zsh

The following files will then exist and can be used:
* ./build/generated/ksp/debug/resources/codebase-map.md
* ./build/generated/ksp/debugUnitTest/resources/codebase-map.md
* ./build/generated/ksp/debugAndroidTest/resources/codebase-map.md

## Testing Guidelines

This project uses specific testing libraries. When writing or updating tests, adhere to these conventions:
* **Assertions:** Use **Kotest Assertions** (`shouldBe`, `shouldThrow`, etc.) instead of JUnit assertions.
* **Property Testing:** Use **Kotest Property Testing** (`checkAll`, `Arb.int()`, etc.) instead of choosing sample values.
* **Mocking:** Use **MockK** (`mockk()`, `every { ... } returns ...`) instead of Mockito.
* **Flows:** Use **Turbine** (`test { ... }`) for testing Kotlin Flows.
